Fluxo Completo de uma Mensagem WhatsApp

Do cliente enviar até receber a resposta — passo a passo com bifurcação Fast Path vs LLM Path.

1
📱 Cliente envia mensagem
Cliente envia texto, áudio ou mídia pelo WhatsApp. A UazAPI recebe e dispara um POST /webhook/uazapi para o backend.
2
🔗 Webhook recebido Roteamento
uaz_webhook.py identifica empresa_id e unidade_id a partir do número WhatsApp da unidade. Valida a assinatura e aplica rate limit por conversa (max 10 req / 10s).
3
📨 Enfileiramento no Redis Stream Assíncrono
Mensagem adicionada ao stream chat_messages:{empresa_id} com campos: conversation_id, content, sender, unidade_id. O webhook retorna 200 OK imediatamente para a UazAPI.
4
🔄 stream_worker.py consome a fila
Worker em background faz XREAD BLOCK 30s no stream. Ao receber a mensagem, chama processar_ia_e_responder() do bot_core.py de forma assíncrona.
5
🧠 Classificação de Intenção ia_processor
classificar_intencao(texto) analisa palavras-chave e padrões para identificar:
  • horario — consulta de horário de funcionamento
  • endereco — localização e endereço da unidade
  • planos — preços, mensalidades, matrículas
  • modalidades — grade de aulas, esportes
  • saudacao — cumprimentos e boas-vindas
  • llm — perguntas abertas → requer IA
⚡ Fast Path (~50ms)
Busca unidade no cache Redis
Carrega dados do PostgreSQL
Formata resposta pronta
Zero custo de token
🤖 LLM Path (~800-2000ms)
Busca histórico (Redis)
RAG: busca semântica FAQ
Monta prompt com contexto
Chama OpenRouter/OpenAI
6
🛡️ Circuit Breaker do LLM LLM Path
Verifica o estado do circuit breaker (cb:llm:state no Redis):
  • CLOSED — chama LLM normalmente com retry (tenacity)
  • HALF_OPEN — testa 1 chamada; se falhar, reabre o circuito
  • OPEN — retorna resposta genérica sem chamar LLM
7
📝 Registro no PostgreSQL Banco
bd_salvar_mensagem_local() — salva mensagem do cliente e resposta da IA.
bd_registrar_evento_funil() — registra mudanças de estado (lead quente, intenção de compra).
registrar_token_usage() — acumula tokens usados por empresa/dia/modelo (fire-and-forget).
8
✍️ Simula digitação
uaz_client.set_presence(number, presence="composing", delay=2000)
Mostra "digitando..." no WhatsApp do cliente enquanto processa.
9
📤 Envio da Resposta
Dependendo da integração ativa da unidade:
  • UazAPI → uaz_client.send_text(fone, texto)
  • Chatwoot → chatwoot_client.enviar_mensagem(account_id, conv_id, texto)
Para mídias: send_media(), para menus: send_menu(), para áudio TTS: send_media(type="myaudio").
10
💾 Atualiza Cache Redis
  • Histórico da conversa (historico:{conversation_id}, TTL 24h)
  • Unidade escolhida (unidade_escolhida:{conversation_id}, TTL 24h)
  • Labels no Chatwoot (se mudou estado do funil)
  • Score do lead atualizado no banco
Tempo total Fast Path: ~50–200ms · LLM Path: ~800–2500ms

Fluxo de Autenticação

Login com JWT de 24 horas, controle de acesso por perfil e registro via convite.

1
🔑 Login
POST /auth/login com body {"username": "email", "password": "senha"}. O backend busca o usuário no PostgreSQL e verifica a senha com bcrypt.
2
🎫 Geração do JWT
Gera token HS256 com JWT_SECRET_KEY válido por 24 horas.
Payload: {"sub": "email", "empresa_id": 5, "perfil": "admin", "exp": ...}.
Retorna: {"access_token": "eyJhbG...", "token_type": "bearer"}
3
💻 Frontend armazena o token
Next.js salva em localStorage.getItem("token"). Todas as requisições enviadas com header: Authorization: Bearer eyJhbG...
4
🛡️ Middleware valida o JWT
Depends(get_current_user_token) no endpoint:
  • Extrai token do header Authorization
  • Verifica assinatura HS256 e expiração (exp)
  • Retorna payload: empresa_id, perfil, sub
  • Se inválido ou expirado → 401 Unauthorized
5
🎭 Controle de Acesso por Perfil
  • admin_master — Acesso total: todas as empresas, criação de tenants
  • admin — Acesso total à própria empresa e unidades
  • operador — Dashboard, conversas e atendimento
  • visualizador — Apenas leitura de métricas
6
📩 Registro via Convite
POST /auth/criar-convite → gera token único e envia e-mail via SMTP.
POST /auth/registrar?token=... → valida o convite, cria usuário com senha bcrypt.
Convites expiram após 48 horas e são de uso único.
7
🚫 Rate Limiting de Login
Máximo 5 tentativas de login / minuto por IP. Endpoints sensíveis de gestão têm rate limiting adicional por empresa_id via Redis: rl:mgmt:{empresa_id}:{tag}:{bucket}.
